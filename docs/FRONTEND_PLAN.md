# 中医问诊与养生平台前端规划

## 1. 前端定位

前端用于把后端 API 变成可操作的浏览器界面。首版目标不是做复杂商业级 UI，而是完成一个清晰、可演示、能联调真实接口的前端应用。

前端范围包括：

- 患者端：浏览内容、注册登录、提交问诊、查看问诊记录、使用 AI 问答。
- 后台端：医生/管理员登录、查看待办问诊、处理问诊、管理文章和药膳、查看 Dashboard、触发导出。

前端会作为后端实战后的第二条学习线加入项目，但不会打乱后端 Day 1-Day 21 的主线。

## 2. 技术栈

首版前端使用：

| 技术 | 用途 |
| --- | --- |
| Vue 3 | 前端应用框架 |
| Vite | 前端构建工具 |
| JavaScript | 首版代码语言，降低学习负担 |
| Vue Router | 页面路由和权限跳转 |
| Pinia | 登录状态、用户信息和全局状态 |
| Axios | 请求后端 API |
| Element Plus | 表单、表格、分页、弹窗、消息提示等 UI 组件 |

选择原因：

- Vue 3 + Vite 是当前 Vue 项目的主流组合。
- Vue Router 是 Vue 官方路由方案。
- Pinia 是 Vue 官方推荐的状态管理方案。
- Element Plus 与 Vue 3 配合成熟，特别适合后台管理页面中的表格、表单和分页。
- 使用 JavaScript 可以减少 TypeScript 类型学习压力，更适合第一次完整实战。

## 3. 项目组织方式

前端使用一个 Vue 应用，不拆成两个独立项目。

```text
frontend/
└─ 一个 Vue 应用
   ├─ 患者端布局
   └─ 后台管理布局
```

原因：

- 登录状态和 token 只维护一套。
- API 请求封装只维护一套。
- 路由权限逻辑更容易理解。
- 构建和部署流程更简单。
- 适合课程项目和个人学习。

不采用：

```text
frontend/
├─ patient-web/
└─ admin-web/
```

因为这会引入两套依赖、两套路由、两套登录状态和两套构建流程，学习负担过高。

## 4. 布局方案

采用双布局：

| 端 | 布局 |
| --- | --- |
| 患者端 | 顶部导航 |
| 后台端 | 左侧边栏 + 顶部用户区域 |

### 4.1 患者端布局

患者端顶部导航示例：

```text
中医问诊与养生平台 | 首页 | 在线问诊 | 养生知识 | 药膳推荐 | AI 问答 | 我的
```

适合场景：

- 浏览首页内容
- 浏览知识文章
- 浏览药膳推荐
- 提交问诊
- 查看个人记录

### 4.2 后台端布局

后台侧边栏示例：

```text
总览
问诊管理
知识文章
药膳管理
数据导出
```

适合场景：

- 高频查看和处理问诊单
- 使用表格、筛选、分页和操作按钮
- 管理内容数据
- 查看统计信息

## 5. 视觉风格

选择“清雅现代中医”风格。

### 5.1 视觉关键词

```text
清雅
克制
现代
可信
有中医辨识度
```

### 5.2 建议色彩

| 角色 | 颜色建议 |
| --- | --- |
| 主色 | 墨绿 |
| 背景 | 白色、浅灰绿 |
| 点缀 | 药材金、浅棕 |
| 文字 | 深灰、深绿 |
| 警示 | Element Plus 默认红/橙，保持清晰 |

### 5.3 页面风格

- 患者端更舒展，适合阅读和填写表单。
- 后台端更紧凑，适合扫描数据和连续操作。
- 不做过重的传统纹样和装饰，避免影响后台效率。
- 首版仅要求电脑端体验，不承诺手机端适配。

## 6. 响应式范围

首版选择：

```text
电脑端优先
```

说明：

- 先完成主要业务流程和接口联调。
- 患者端和后台端都按桌面浏览器设计。
- 手机端适配作为后续扩展任务。
- 编写布局时仍应保留清晰组件边界，避免后续完全重写。

后续可扩展：

- 患者端移动适配
- 后台端宽屏表格优化
- 手机端简化问诊表单

## 7. 权限与访问规则

### 7.1 游客权限

游客可以访问：

- 首页
- 中医知识文章列表
- 中医知识文章详情
- 药膳推荐列表
- 药膳推荐详情

游客不能访问：

- 提交问诊
- 我的问诊记录
- AI 问答
- 后台管理页面

### 7.2 患者权限

患者登录后可以访问：

- 提交问诊
- 我的问诊记录
- AI 问答
- 个人信息相关页面

### 7.3 医生/管理员权限

医生和管理员共用一套后台界面。

后台始终需要登录。

医生和管理员可访问：

- 后台工作台
- 问诊管理
- 知识文章管理
- 药膳管理
- Dashboard 统计
- 数据导出

如果后续出现管理员专属能力，再通过角色控制隐藏或禁用入口。

## 8. 登录入口

登录入口分离：

```text
/login/patient   患者登录
/login/admin     医生与管理员登录
/register        患者注册
```

设计原则：

- 患者端不主动展示后台登录入口。
- 后台登录通过独立地址访问。
- 登录成功后根据角色跳转：
  - 患者 -> 患者首页或我的问诊
  - 医生/管理员 -> 后台工作台

## 10. 页面范围

### 10.1 患者端页面

| 页面 | 主要能力 |
| --- | --- |
| 首页 | 突出在线问诊入口，展示知识、药膳、AI 快捷入口 |
| 患者登录 | 输入用户名和密码，调用患者登录接口 |
| 患者注册 | 输入用户名、密码、昵称、手机号 |
| 提交问诊 | 填写问诊表单并提交 |
| 我的问诊 | 查看当前患者提交过的问诊记录 |
| 知识文章列表 | 浏览已发布文章 |
| 知识文章详情 | 阅读完整文章 |
| 药膳推荐列表 | 浏览已发布药膳，支持季节和体质筛选 |
| 药膳推荐详情 | 查看食材、步骤和适宜人群 |
| AI 问答 | 输入问题，展示 AI 或 fallback 回答 |
| 个人中心 | 显示当前登录信息，提供退出登录 |

### 10.2 后台页面

| 页面 | 主要能力 |
| --- | --- |
| 后台登录 | 医生/管理员登录 |
| 后台工作台 | 待办问诊优先，展示紧急和待接诊数据 |
| 问诊管理 | 筛选、分页、查看和更新问诊单 |
| 知识文章管理 | 文章列表、新增、编辑、删除、发布状态 |
| 药膳管理 | 药膳列表、新增、编辑、删除、发布状态 |
| 数据导出 | 触发问诊单 CSV 导出 |

## 11. 首页设计决策

### 11.1 患者首页

选择：

```text
问诊入口优先
```

第一屏重点：

- 简洁说明平台用途
- 明确的“开始在线问诊”按钮
- 知识文章入口
- 药膳推荐入口
- AI 问答入口

原因：

- 问诊单是当前项目的核心业务数据。
- 便于后端 Day 8-Day 10 完成后进行真实联调。
- 公开内容仍可以作为辅助入口展示。

### 11.2 后台首页

选择：

```text
待办问诊优先
```

第一屏重点：

- 待接诊数量
- 紧急问诊数量
- 接诊中数量
- 今日新增数量
- 优先处理列表
- 简要趋势或状态统计

原因：

- 医生登录后的第一任务是处理问诊。
- 能把后端提醒规则真正体现在界面中。
- Dashboard 数据不是纯展示，而是服务于处理优先级。

## 12. API 依赖

前端开发依赖后端接口逐步完成。

| 前端能力 | 后端依赖 |
| --- | --- |
| 患者登录 | `/api/auth/login/patient` |
| 后台登录 | `/api/auth/login/admin` |
| 患者注册 | `/api/auth/register` |
| 提交问诊 | `POST /api/patient/consultation` |
| 我的问诊 | `GET /api/patient/consultation/my` |
| 知识文章列表 | `GET /api/patient/knowledge` |
| 药膳推荐列表 | `GET /api/patient/recipe` |
| 后台 Dashboard | `GET /api/admin/dashboard` |
| 后台问诊管理 | `GET /api/admin/consultation`、`PUT /api/admin/consultation/{id}` |
| 文章管理 | `/api/admin/knowledge[/{id}]` |
| 药膳管理 | `/api/admin/recipe[/{id}]` |
| CSV 导出 | `GET /api/admin/export/consultations` |
| AI 问答 | `POST /api/patient/ai/question` |

## 13. 前端开发阶段

前端不从项目一开始就写。建议在后端 Day 10 完成后正式介入。

### 前置条件

开始前端前，后端应至少完成：

- Day 1-Day 6：认证、权限、统一响应
- Day 8-Day 10：问诊主流程和患者端核心接口

### Frontend Phase 1：初始化与基础架构

目标：

- 初始化 Vite + Vue 项目
- 安装 Vue Router、Pinia、Axios、Element Plus
- 建立目录结构
- 建立患者端布局和后台布局
- 实现路由守卫
- 实现 Axios 请求封装
- 实现 token 存储和退出登录

### Frontend Phase 2：登录注册与权限

目标：

- 患者登录页
- 后台登录页
- 患者注册页
- 登录成功跳转
- 未登录拦截
- 角色不匹配拦截

### Frontend Phase 3：患者端核心流程

目标：

- 患者首页
- 提交问诊表单
- 我的问诊记录
- 问诊状态展示

### Frontend Phase 4：后台问诊工作台

目标：

- 后台工作台
- 待办问诊列表
- 问诊管理表格
- 问诊筛选
- 问诊状态更新

### Frontend Phase 5：内容与 AI

目标：

- 知识文章列表和详情
- 药膳推荐列表和详情
- AI 问答页面
- 后台文章管理
- 后台药膳管理

### Frontend Phase 6：联调、构建与演示

目标：

- 前后端联调
- 错误提示优化
- 空状态和加载状态
- 构建前端
- 准备演示流程

## 14. Git/GitHub 规划

前端进入时，建议创建独立分支：

```text
feature/frontend-foundation
```

后续分支建议：

| 阶段 | 分支 |
| --- | --- |
| 前端基础架构 | `feature/frontend-foundation` |
| 登录注册 | `feature/frontend-auth` |
| 患者端页面 | `feature/frontend-patient` |
| 后台问诊管理 | `feature/frontend-admin-consultation` |
| 内容和 AI 页面 | `feature/frontend-content-ai` |
| 前后端联调 | `test/fullstack-integration` |

前端每个阶段都应：

1. 先确认对应后端接口是否可用。
2. 实现页面。
3. 运行前端开发服务器验证。
4. 检查浏览器控制台错误。
5. 提交前查看 Git diff。
6. 根据阶段决定是否创建 PR。
