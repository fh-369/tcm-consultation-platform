## Git 安全撤销知识

### 场景一：修改尚未提交

此时改动只存在于工作树。VS Code 可以针对单个明确文件选择 **Discard Changes**。

影响：

- 只撤销该文件尚未提交的修改。
- 已提交历史不会改变。
- 撤销后内容通常难以恢复，因此操作前必须确认文件差异。

### 场景二：提交已创建并推送

已共享的提交应使用 `git revert <commit>` 创建一个新的反向提交。

影响：

- 原提交仍保留在历史中。
- 新提交撤销原提交造成的内容变化。
- 不重写远程历史，适合多人协作和已经推送的分支。

### 场景三：提交只存在于本地

可以根据是否需要保留文件修改选择恢复方案，但必须先确认提交是否已推送、工作树是否干净，以及哪些内容仍需保留。

## 安全原则

- 不默认执行 `git reset --hard`。
- 不使用强制推送重写共享历史。
- 不把删除分支误认为删除提交内容；已合并提交仍由 `main` 和 PR 历史保存。
- 撤销前先确认错误位于工作树、暂存区、本地提交还是远程提交。
