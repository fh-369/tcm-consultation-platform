# 结合项目实战引导的 Git 与 GitHub 伴学指南

## 一、这份文件解决什么问题

这份文件不是一个独立的示例项目教程，也不会自行规定项目功能。

它需要与另一份“项目实战引导文件”一起交给 AI 使用。项目实战引导文件负责说明项目要做什么、分成哪些阶段、每一步实现什么功能；本文件负责让 AI 把 Git 和 GitHub 学习自然嵌入这些项目步骤中。

例如，项目实战引导可能包含：

```text
步骤 1：初始化项目
步骤 2：完成首页
步骤 3：增加登录功能
步骤 4：修复登录错误
步骤 5：部署项目
```

AI 不应另外创建一套脱离项目的 Git 练习，而应把这些步骤转化为真实的 Git/GitHub 实践：

```text
初始化项目       → 初始化 Git 仓库并创建第一次提交
完成首页         → 创建功能分支、分阶段提交、合并
增加登录功能     → 使用独立分支或工作树开发
修复登录错误     → 创建修复分支并使用 Pull Request
部署项目         → 创建版本标签并检查发布记录
```

## 二、开始使用

开始一个新项目时，同时向 AI 提供：

1. 项目实战引导文件
2. 本文件
3. 项目代码所在目录

然后发送：

> 请先完整阅读项目实战引导文件和《结合项目实战引导的 Git 与 GitHub 伴学指南》。  
> 项目实战引导决定项目功能、实现顺序和验收目标；Git 伴学指南决定每一步如何练习 Git 和 GitHub。  
>   
> 请先分析项目引导并生成“项目步骤与 Git 学习点映射表”，不要立即修改代码。等我确认映射表后，再从项目引导的第一个未完成步骤开始。  
>   
> 每个需要进行 Git 操作的节点，都让我在以下方式中二选一：  
> 1. `Codex`：我向你描述目标，由你解释并执行 Git 操作；  
> 2. `VS Code`：你告诉我在 VS Code 中如何操作，由我亲手完成。  
>   
> 每次只推进项目引导中的一个小步骤。不要为了教学而打乱项目顺序，也不要一次性完成多个步骤。

## 三、两份文件的职责边界

### 项目实战引导文件负责

- 项目目标和业务功能
- 技术栈和项目结构
- 功能实现顺序
- 每个阶段的代码任务
- 功能验收标准

### 本文件负责

- 什么时候创建 Git 仓库
- 什么时候创建、切换和合并分支
- 什么时候创建工作树
- 什么时候暂存和提交
- 如何编写提交消息
- 什么时候推送到 GitHub
- 什么时候创建 Pull Request
- 如何处理冲突、回退和版本标签
- 如何在 Codex 与 VS Code 操作方式之间二选一

### AI 不得做的事情

- 不得用本文件中的示例任务替换项目实战引导中的真实任务。
- 不得为了展示 Git 命令，制造与项目无关的文件或功能。
- 不得跳过项目实战引导中的步骤。
- 不得在用户没有确认时一次完成多个功能步骤。
- 不得把“代码实现完成”和“Git 操作完成”混为一件事。
- 不得默认提交、合并、推送、删除分支或移除工作树。

## 四、AI 首次读取文件后的任务

AI 必须先读取项目实战引导，并输出一份映射表。

推荐格式：

| 项目步骤 | 功能目标 | 建议分支 | 建议 Git 操作 | Git 学习点 | 是否适合工作树 | GitHub 节点 |
|---|---|---|---|---|---|---|
| 步骤 1 | 初始化项目 | `main` | 初始化仓库、第一次提交 | 工作目录、暂存区、commit | 否 | 创建远程仓库 |
| 步骤 2 | 完成首页 | `feature/homepage` | 创建分支、分阶段提交、合并 | 分支与 merge | 否 | 可创建 PR |
| 步骤 3 | 登录功能 | `feature/login` | 创建分支或工作树、推送 | 工作树、远程分支 | 是 | 创建 PR |
| 步骤 4 | 修复错误 | `fix/login-error` | 修复提交、revert 或合并 | 修复分支、安全撤销 | 是 | PR 审查 |

映射表必须满足：

- 项目步骤名称来自真实的项目实战引导。
- Git 学习点应匹配当前步骤，不应生硬加入。
- 简单连续的小步骤可以在同一个分支中完成。
- 独立功能、实验性修改和错误修复优先使用独立分支。
- 只有确实需要同时保留多个开发现场时，才建议额外工作树。
- GitHub 操作应逐步引入，不要在第一步一次讲完。

用户确认映射表后，AI 才能开始执行项目步骤。

## 五、每个项目步骤的标准教学循环

AI 对项目引导中的每个步骤，都应按以下顺序进行。

### 1. 定位当前步骤

AI 明确说明：

```text
项目引导位置：
当前阶段：
当前步骤：
本步功能目标：
本步验收标准：
```

### 2. 检查当前 Git 状态

开始修改前，AI 应检查或指导用户检查：

```powershell
git status
git branch --show-current
git log --oneline --graph --all --decorate -n 15
git worktree list
git remote -v
```

并解释：

- 当前位于哪个工作树和分支
- 是否存在未提交修改
- 当前步骤是否应该继续使用现有分支
- 是否应该创建新分支或工作树
- 是否需要先同步 GitHub 远程变化

### 3. 制定本步 Git 计划

AI 在开始写代码前给出简短计划：

```text
本步代码任务：
本步使用的分支：
计划产生的 commit：
是否需要推送：
是否需要 Pull Request：
完成后如何验收：
```

计划产生的 commit 应按照真实开发进度拆分，而不是为了数量机械拆分。

### 4. 选择 Git 操作方式

每当需要进行 Git 操作时，AI 必须让用户二选一：

```text
这次 Git 操作请选择：
1. Codex：你告诉我目标，我解释并执行
2. VS Code：我给出界面步骤，你亲手操作
```

每次只选择一种方式，不重复执行相同操作。

### 5. 完成当前代码任务

AI 按项目实战引导完成或指导用户完成代码任务。

Git 教学不能遮盖项目本身的验收。提交前必须先确认：

- 当前小步骤功能已经完成
- 相关测试或检查已经通过
- 没有明显无关修改
- 用户理解准备提交的内容

### 6. 审查修改

提交前，AI 必须指导用户查看差异：

```powershell
git status
git diff
git diff --staged
```

AI 应解释：

- 哪些文件发生变化
- 哪些变化属于当前步骤
- 是否混入其他任务修改
- 当前应该暂存哪些文件

### 7. 暂存和提交

AI 不应默认使用 `git add .`。初学阶段优先按明确文件暂存，让用户知道每个文件为什么进入提交。

提交消息应说明本次完成的真实工作，例如：

```text
Add login form validation
Fix empty username error
Document local setup steps
```

避免没有含义的提交消息：

```text
update
change
fix
完成
```

### 8. 验证提交结果

提交后，AI 应检查：

```powershell
git status
git log --oneline --graph --all --decorate -n 15
```

并让用户回答：

1. 新 commit 保存了什么？
2. 当前分支指向哪里？
3. 其他分支或工作树的文件是否自动变化？

### 9. 判断是否推送或合并

当前小步骤完成后，AI 应结合项目引导判断：

- 继续留在当前分支完成下一小步
- 推送当前分支作为备份
- 创建 Pull Request
- 合并到 `main`
- 等当前阶段完成后再合并

AI 必须解释理由，不能机械地每次提交后都合并。

### 10. 更新学习进度

AI 在每个项目步骤结束后简短记录：

```text
项目进度：
本步完成的功能：
本步产生的 commit：
本步使用的 Git/GitHub 操作：
已经掌握的概念：
仍需复习的概念：
下一项目步骤：
```

## 六、Codex 与 VS Code 二选一规则

### 选择 Codex 时

用户使用自然语言描述目标，不需要自己输入完整 Git 命令。

推荐表达：

> 请检查当前状态，并为这个功能创建合适的分支。执行前先告诉我分支名和原因。

> 请只暂存属于当前项目步骤的文件，列出准备暂存的文件，等我确认后再提交。

> 请展示当前分支与 `main` 的差异，并判断现在是否适合创建 Pull Request。

> 请将这个分支合并到 `main`。执行前说明可能产生的结果和风险。

Codex 必须：

- 操作前解释准备执行什么
- 检查工作树和分支状态
- 明确列出将修改、暂存、提交、合并或推送的内容
- 操作后展示并解释结果
- 不得顺便执行未经要求的后续操作

### 选择 VS Code 时

AI 应指导用户通过界面亲手完成，不应替用户执行同一个 Git 操作。

常见操作入口：

| 目标 | VS Code 操作 |
|---|---|
| 查看修改 | 左侧“源代码管理”，点击“更改”中的文件 |
| 暂存文件 | 点击文件旁边的 `+` |
| 取消暂存 | 点击“暂存的更改”中文件旁边的 `-` |
| 创建提交 | 输入提交消息，点击“提交” |
| 创建或切换分支 | 点击左下角分支名称 |
| 合并分支 | `Ctrl+Shift+P`，运行 `Git: Merge Branch...` |
| 发布分支 | 点击“发布分支” |
| 同步远程 | 使用同步按钮，并在操作前确认方向 |
| 查看提交图 | 使用 Source Control Graph 或已有 Git 图形扩展 |
| 创建 PR | 使用 GitHub Pull Requests 扩展或 GitHub 网页 |

AI 必须：

- 给出准确的点击路径
- 说明用户完成后应该看到什么
- 等待用户完成并反馈结果
- 根据用户截图或界面版本调整指导
- 通过 Git 状态或提交图验证操作结果

## 七、如何把项目步骤转换为分支策略

AI 应根据项目引导中的任务性质选择分支，而不是每个微小步骤都创建分支。

### 继续使用当前分支

适用于：

- 同一功能内部的连续小步骤
- 文档或代码的小幅补充
- 当前分支尚未完成的同一任务

### 创建功能分支

适用于：

- 新增一项相对独立的功能
- 项目引导开始新的功能阶段
- 修改需要单独审查

命名示例：

```text
feature/login
feature/search
feature/user-profile
```

### 创建修复分支

适用于：

- 修复已经存在的错误
- 当前主线需要独立修复

命名示例：

```text
fix/login-validation
fix/payment-timeout
```

### 创建实验分支

适用于：

- 项目引导要求尝试不同实现方案
- 结果可能被放弃

命名示例：

```text
experiment/new-layout
experiment/cache-strategy
```

### 创建工作树

只有出现以下情况时才建议：

- 当前功能开发到一半，需要同时修复 `main` 上的紧急问题
- 需要同时运行或比较两个分支
- Codex 与用户需要在不同任务上并行工作
- 需要保留当前未提交现场，同时处理另一任务

工作树不是业务模块，也不是分支替代品。它是让不同分支同时拥有独立编辑目录的工具。

## 八、如何把项目步骤转换为 commit

一个项目步骤不一定只对应一个 commit，也不应被切成大量没有意义的 commit。

### 适合提交的节点

- 一个可解释的小功能完成并通过检查
- 一项独立错误修复完成
- 一组相关重构完成且行为未改变
- 文档与配置完成一个明确目的

### 不适合立即提交的节点

- 代码无法运行且没有教学目的
- 当前修改与提交消息无法清晰对应
- 混入多个不相关任务
- 尚未查看修改内容

### 项目步骤与 commit 示例

假设项目引导中的一个步骤是“实现登录功能”，可能拆分为：

```text
Add login page layout
Add login form validation
Connect login form to API
Handle login API errors
```

是否拆分取决于项目实际复杂度。AI 应解释拆分理由，并让用户看到每次提交如何形成可阅读的开发历史。

## 九、GitHub 应如何嵌入项目步骤

GitHub 操作应服务于真实项目协作，而不是独立练习。

### 建议引入顺序

1. 项目初始化后创建 GitHub 远程仓库。
2. 完成第一次稳定提交后推送 `main`。
3. 第一个功能分支完成后发布分支。
4. 使用 Pull Request 审查该功能。
5. 合并 PR 后使用 fetch 或 pull 同步本地状态。
6. 项目达到阶段里程碑时创建 tag 或 release。

### Pull Request 必须包含

- 本次完成了哪个项目引导步骤
- 修改目的
- 主要文件和行为变化
- 验证方式
- 已知风险或未完成事项

### GitHub 操作前必须区分

```text
commit：在本地保存代码快照
push：把本地提交发送到 GitHub
Pull Request：请求审查并合并分支
merge：把两条开发历史整合
fetch：获取远程信息，不直接修改当前文件
pull：获取远程信息并整合到当前分支
```

## 十、项目引导中常见场景的处理方式

### 场景 1：项目引导要求初始化项目

AI 应结合 Git 完成：

- 检查是否已有仓库
- 初始化 Git
- 检查 `.gitignore`
- 创建第一次清晰提交
- 根据用户意愿连接 GitHub

### 场景 2：项目引导进入新功能

AI 应先判断：

- 是否创建新功能分支
- 是否需要工作树
- 预计产生几个合理 commit
- 功能完成后是否创建 PR

### 场景 3：项目引导要求修改已有功能

AI 应先展示当前历史与差异，帮助用户判断：

- 修改属于当前分支还是新分支
- 是否存在未提交修改
- 是否会影响其他开发任务

### 场景 4：项目引导要求修复错误

AI 应创建或建议修复分支，并在修复后：

- 运行验证
- 创建明确修复提交
- 通过 PR 或 merge 整合
- 必要时讲解 `git revert`

### 场景 5：项目引导要求尝试多个方案

AI 可以建议：

- 为不同方案创建实验分支
- 使用工作树同时打开方案
- 比较差异后保留选定方案

不得未经用户确认删除未采用方案的分支或工作树。

### 场景 6：项目引导完成一个阶段

AI 应进行阶段检查：

- 工作目录是否干净
- 阶段功能是否通过验收
- 分支是否已经正确合并
- GitHub 是否已经同步
- 是否需要创建版本标签
- 是否存在可以安全清理的分支或工作树

清理操作必须单独确认。

## 十一、冲突和错误必须成为教学内容

出现冲突、命令失败或 detached HEAD 时，AI 不应直接绕过。

AI 应依次说明：

1. 当前发生了什么
2. 为什么发生
3. 哪些修改仍然安全存在
4. 有哪些解决方案
5. 每种方案会改变什么
6. 推荐方案及理由

合并冲突时，AI 不得直接替用户决定业务内容。应先解释冲突双方，再让用户决定最终结果。

---

## 十二、安全规则

AI 必须遵守：

1. 不批量删除文件或目录。
2. 删除文件时，只能处理一个明确路径，并先获得确认。
3. 不使用 `del /s`、`rd /s`、`rmdir /s`、`Remove-Item -Recurse` 或 `rm -rf`。
4. 不在未说明风险时执行 `reset --hard`、rebase、强制推送或历史重写。
5. 不删除仍有未提交修改的分支或工作树。
6. 不直接删除工作树目录。
7. 不默认使用 `git add .` 暂存所有修改。
8. 不自动合并、推送、删除分支或清理工作树。
9. 发现与当前任务无关的修改时，不覆盖、不撤销，先说明情况。
10. 每次重要 Git 操作前后都检查状态。

```markdown
- 项目引导文件：
- 当前阶段：
- 当前步骤：
- 当前工作树：

- 已掌握：
- 需要复习：

## 下一步

- 项目任务：
- 建议 Git 操作：
- 等待用户选择：Codex / VS Code
```

## 十四、每步结束时的验收问题

AI 每完成一个项目步骤，至少询问：

1. 本步项目功能完成了什么？
2. 本步产生了哪些 commit？为什么这样拆分？
3. 当前分支与 `main` 有什么区别？
4. 本步修改是否已经推送到 GitHub？
5. push、PR 和 merge 中哪些已经发生，哪些还没有？
6. 如果现在切换分支或工作树，哪些内容会变化，哪些不会？

AI 应根据用户回答纠正误解，再继续项目引导的下一步骤。

## 十五、最终验收标准

完成整个项目实战引导后，用户应不仅完成项目，还应能够：

- 根据新任务判断是否需要创建分支
- 根据并行开发需求判断是否需要工作树
- 在 Codex 中用自然语言安全地请求 Git 操作
- 在 VS Code 中亲手查看、暂存、提交和切换分支
- 将一个项目步骤拆分成合理 commit
- 阅读分支图并解释开发历史
- 创建、审查和合并 Pull Request
- 区分 commit、push、PR、merge、fetch 和 pull
- 处理合并冲突
- 使用安全方式撤销错误修改
- 在操作前识别可能丢失代码的风险

最终工作流程应接近：

```text
读取项目引导中的下一步骤
→ 检查当前 Git 状态
```
